# Harness Engineering 원칙 (Harness Engineering Principles)

본 프로젝트는 "완성된 기존 레이아웃"을 보존하면서 새로운 기능이나 변경 사항을 안전하게 통합하기 위해 **하네스 엔지니어링(Harness Engineering)** 원칙을 따릅니다.

## 1. 개요
Harness(하네스)란 기존 시스템을 안전하게 감싸고, 외부의 변화가 내부의 안정성에 영향을 미치지 않도록 보호하는 래퍼(Wrapper) 구조를 의미합니다. 프론트엔드 환경에서는 기존 렌더링 코드나 스타일(CSS)에 대한 직접적인 접근(Write)을 제한하고, 신규 요소를 독립적인 컨테이너 안에서만 동작하도록 격리하는 기법입니다.

## 2. 주요 원칙

### 1) 격리 (Isolation)
- **독립된 컴포넌트화**: 기존에 작성된 복잡한 로직(예: Sales Team 대시보드)을 별도의 컴포넌트(예: `SalesTeamSection.vue`)로 분리하여 메인 뷰 컴포넌트(`DashboardView.vue`)의 복잡도를 낮춥니다.
- **스타일 캡슐화**: 분리된 컴포넌트에는 `<style scoped>`를 적용하여, 이곳의 CSS 클래스의 변경이나 추가가 다른 영역에 영향을 주지 않도록 완벽히 격리합니다.

### 2) 무수정 보존 (No-Write Policy for Completed Regions)
- '완성된 영역'으로 정의된 코드 및 스타일은 절대 수정하지 않습니다.
- 예: 폰트 사이즈, 마진 설정, 특정 차트의 애니메이션 옵션 등은 원본 코드를 그대로 보존하여, 시각적 회귀(Visual Regression)를 방지합니다.

### 3) 독립 컨텍스트 구축 (Independent Context)
- **신규 기능 추가 시**: 신규 작업이 필요한 경우, 완성된 영역 외의 [상단/하단/우측] 빈 공간에 독립적인 래퍼 컨테이너(Harness)를 생성한 뒤 그 안에서 작업합니다.
- 데이터 생성 로직 상호 간섭 방지: 예를 들어, 개발팀 도표의 UV와 PV 데이터는 각각 독립적인 로직을 통해서만 계산되어야 하며, 서로 종속되지 않게 설계해야 합니다.

## 3. 적용 사례: 대시보드 리팩토링
- **영업팀 (SalesTeamSection)**: 기존 완성된 영역에 해당하는 좌측 도표와 우측 지표 카드 코드를 그대로 추출하여 `SalesTeamSection.vue`에 담습니다. (완성 영역 지정 완료 - 수정 금지)
- **개발팀 (DevTeamSection)**: `DevTeamSection.vue` 및 해당 컴포넌트가 렌더링하는 영역 전체(코드 및 스타일(CSS) 포함)는 **'완성 영역'**으로 지정되었습니다. 기존 레이아웃과 디자인(폰트, 색상, 애니메이션 등)을 깨뜨리지 않기 위해 해당 영역에 대한 모든 수정('Write' 접근)을 엄격히 금지합니다. 향후 신규 추가되는 기능은 완성 영역 외의 상단/하단/우측 빈 공간 등 완전히 독립적인 컨테이너 내부에서만 동작하도록 하네스 원칙을 준수해야 합니다.
- **영상마케팅팀 (VideoTeamSection)**: 기존 대시보드 뷰에서 분리하여 `VideoTeamSection.vue`를 생성했습니다. 상/하단 2개의 도표 구조와 3열 지표 분석 카드를 채택하였으며, 내부의 로컬 뷰 컴포넌트를 조작할 때 `:deep()` 선택자를 사용해 전역 범위를 오염시키지 않도록 캡슐화했습니다. 이 역시 '완성 영역'으로 간주합니다.
- **KPI 파이프라인 (KpiPipelineSection)**: `KpiPipelineSection.vue`를 신규 생성하여 전체 대시보드의 최하단에 배치했습니다. 기존 완성 영역 컴포넌트들을 전혀 수정하지 않고, 독립된 Vue State와 Custom CSS(flexbox 및 keyframes) 만으로 구현된 반응형 데이터 플로우 컴포넌트입니다.

---
정리: 하네스 엔지니어링은 "기존 코드를 건드리지 않고, 안전한 껍데기(Harness)를 씌워 시스템을 확장하는 것"이 핵심입니다.
